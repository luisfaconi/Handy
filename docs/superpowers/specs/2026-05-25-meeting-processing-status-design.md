# Meeting Mode Processing Status — Design

**Date:** 2026-05-25
**Scope:** Meeting mode only (Windows). Adds a persistent `processing_status` field to history entries, with a visible badge on the history card showing `Starting → Processing → Completed`.

---

## Problem

When using slower Whisper models, meeting mode transcriptions take noticeable time to process. The history panel gives no indication of whether a segment is still being transcribed or has finished. Users have no feedback during the processing window.

---

## Goals

- Show three states on meeting history cards: **Iniciando** (starting), **Processando** (processing), **Finalizado** (completed)
- Status persists in the database — old entries show "Finalizado" permanently
- Badge is visible only on `entry_type = "meeting"` entries

---

## Data Model

### DB Migration (migration 6, `history.rs`)

```sql
ALTER TABLE transcription_history
ADD COLUMN processing_status TEXT NOT NULL DEFAULT 'completed';
```

Existing entries automatically receive `'completed'`.

### Rust Enum

```rust
#[derive(Clone, Debug, Serialize, Deserialize, Type, PartialEq, Eq)]
#[serde(rename_all = "snake_case")]
pub enum ProcessingStatus {
    Starting,
    Processing,
    Completed,
}
```

### `HistoryEntry` struct

Add field:
```rust
pub processing_status: ProcessingStatus,
```

`bindings.ts` is auto-generated by tauri-specta — the TypeScript type `"starting" | "processing" | "completed"` appears automatically on next build.

### New `HistoryManager` method

```rust
pub fn update_processing_status(&self, id: i64, status: ProcessingStatus) -> Result<()>
```

Updates only the `processing_status` column and emits `HistoryUpdatePayload::Updated { entry }` so the frontend receives the change reactively.

`save_entry` gains a `processing_status: ProcessingStatus` parameter.

---

## Backend Flow (`audio.rs`)

### Current behavior
The history entry is created lazily — only after the first audio segment is transcribed inside the worker thread.

### New behavior

**Before spawning the worker** (main thread, in `begin_meeting_segment`):
```
PTT start
  → hm.save_entry("", ProcessingStatus::Starting)
  → returns placeholder_id
  → placeholder_id is moved into the worker closure
```

**Inside the worker thread:**
```
1. First segment transcribed:
   → hm.update_transcription(placeholder_id, text)
   → hm.update_processing_status(placeholder_id, Processing)

2. Each subsequent segment:
   → hm.update_transcription(placeholder_id, full_text)
   (status unchanged, stays Processing)

3. Channel drained (PTT stop, worker finishes):
   → hm.update_transcription(placeholder_id, full_text_final)
   → hm.update_processing_status(placeholder_id, Completed)
```

The frontend already listens to `HistoryUpdatePayload::Added` and `::Updated` — no new Tauri events are needed. The `Added` payload will carry `processing_status: starting` and subsequent `Updated` payloads will carry the new status.

## Frontend

### Badge component

Rendered only when `entry.entry_type === "meeting"`. Positioned as a small chip in the history card (top area, consistent with existing card layout).

| `processing_status` | Color | Icon | i18n key |
|---|---|---|---|
| `starting` | Yellow/amber | Spinner | `history.status.starting` |
| `processing` | Blue | Spinner | `history.status.processing` |
| `completed` | Green | None | `history.status.completed` |

### i18n keys (`src/i18n/locales/en/translation.json`)

```json
"history": {
  "status": {
    "starting": "Starting...",
    "processing": "Processing...",
    "completed": "Completed"
  }
}
```

All other locale files receive the same keys (values in English as fallback; native translations can follow separately).

## Edge Cases

- **App crash mid-session:** Entry remains in `starting` or `processing` state permanently. These states are honest — the transcription was never completed. The badge will show "Iniciando" or "Processando" indefinitely, which accurately reflects what happened.
- **Old entries:** Migration default `'completed'` means all pre-existing meeting entries show "Finalizado" without any manual backfill.
- **Non-meeting entries:** `processing_status` column exists for all rows (SQL constraint requires it), but the frontend badge is only rendered for `entry_type = "meeting"`.
